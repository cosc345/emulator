COSC345 Assignment 2: Draft System – Report

Reflection on Experience

The experience of creating the draft system for our health app has been very enlightening. We encountered many challenges that were not previously foreseen, such as schedule conflicts and technical difficulties.

Firstly we had the challenge of scheduling group meetings where everyone could be present. More often than not, we had members who were absent due to commitments to other courses, work or health issues. Consequently, we used text messaging, social media and GitHub as a solution to our communication problems. Whilst this solution was mostly effective, there were times where we found that slow responses or a lack of response, delayed the decision-making process.

Furthermore, we also had scheduling conflicts with assessments from other courses. The plan that we originally drafted did not account for the time we would need to prepare for our other assessments and as a result, we ended up over-estimating the amount of time we had on hand to learn our programming languages and create this draft system. This also means that we were unable to keep to the roles that we originally had planned for each member, as other members picked up the slack when each member was busy with other commitments. As a result we could only complete the heart rate monitor for testing and the clock, GPS locator, mock collection/database and medicine notification remain unfinished.

Another challenge faced by our group was our weak technical background. Every member in our group is inexperienced with JavaScript and had little to no prior experience with HTML5. While we initially anticipated this problem and hence designated time to learning both programming languages, our inexperience meant that we spent a lot more time researching for solutions than we had previously planned for. Some of the technical difficulties that we were unable to solve for the draft system include:

  • Testing with QUnit – we came to the realization after finishing the code for the draft system that we will need to alter the coding for the emulator in order to be able to test using QUnit.
  • Interface – this will require more polishing, as at the moment it is just a logo that is then followed by plain text that says ‘heart rate’ and ‘heart’. We plan on making this more aesthetically pleasing in the next assignment.
  • Heart rate monitor of our app – currently, the canvas will not clear and update to show the new heart rate. As a result, we need to re-run the script in order to create a new app object for that script so that the screen can continuously update with the new heart rate. Our planned improvement for this problem is to use a method that will not require the re-running of the script.

    • Changes Made

    The biggest change we made when creating this draft system is changing our choice of IDE. Originally, we had planned to use Eclipse as our members were more familiar with it but when we started programming our app, we quickly realized the limitations in using Eclipse. As a result, we decided to change to using NetBeans. NetBeans made pushing and pulling from GitHub a lot more efficient as well as offering the advantageous features provided by Eclipse.

    Due to the limited amount of the app that we have finished for testing, we have yet to make any other changes to our original plan of what features we want in our app. However we do anticipate that we will be making a lot of changes to our draft system, such as those mentioned above, as we work towards the finished system.

    Testing

    The main method of testing we used for our draft system is console.log. We created a random number generator that is used to imitate the heartbeat and we used console.log to print out messages such as “….dangerous heartbeat, call 111 immediately”, when an extreme heart rate value was reached.

    After the lecture on QUnit, we attempted to create test files using QUnit. However as stated above, because of the approach that we have taken to code the emulator, QUnit testing currently creates problems for us. As a result, we plan on altering our emulator in order to allow for QUnit testing.

    Evaluation

    Overall we feel like we have made a pretty good attempt at the draft system, especially considering our circumstances with mid-terms and assessments in our other courses, as well as our inability to meet together as a whole group more than once during this period. We have definitely learnt that we over-estimated our abilities to solve unforeseen technical difficulties as well as the amount of time we had on hand to program this draft system. There is also a lot more to improve on, in terms of communication between members to ensure more efficient cooperation and decision-making and also following the plan that we create. We hope to take what we having learned from assignment 2 to improve the quality of our finished system.
